# PROJECT.md

## Identity

A GitHub "special repository" monorepo. GitHub displays the `README.md` of a repository whose
name matches the owner's username directly on their profile page. This project extends that idea:
on every push to `main` it auto-generates `README.md` from live GitHub API data, deploys a full
resume application to GitHub Pages, and maintains a shared Angular component library used by both
outputs. The entire system is driven by a single source of truth: `packages/profile/src/profile.json`.

Top-level keys in `profile.json`: `name` | `headline` | `summary` | `location` | `links` | `experience` | `education` | `certifications` | `projects` | `skills` | `languages`

> **Do NOT edit manually**: `README.md` (auto-generated by CD), `artifacts/` (build outputs), `pnpm-lock.yaml` (auto-managed by pnpm).

## Data Flow

This is the most commonly lost context across sessions. Understand it before touching any workspace.

```text
packages/profile/src/profile.json      ← THE SINGLE SOURCE OF TRUTH
          │
          ▼
packages/profile/src/schema.ts         ← Zod validation (build fails on invalid data)
          │
          ▼
packages/profile/src/data.ts           ← Exports PUBLIC_PROFILE / PRIVATE_PROFILE (deep-frozen)
          │
          ├── apps/resume/              → Angular Signals store → Angular components render sections
          ├── apps/readme/              → NestJS service → markdown templates → README.md
          ├── packages/design-system/   → Storybook mock data
          └── quality/*/                → test assertions
```

To update any data shown in the resume or README: edit `profile.json` only. The schema in
`schema.ts` is enforced at build time — a field violation produces a descriptive error pointing
to the offending key.

## Key Operations

| Command                     | What it does                                                         |
| --------------------------- | -------------------------------------------------------------------- |
| `pnpm install`              | Install all workspace dependencies                                   |
| `pnpm run serve:resume`     | Dev server for `@vh/resume` at `http://localhost:4200`               |
| `pnpm run storybook`        | Launch Storybook for `@vh/design-system`                             |
| `pnpm run generate:readme`  | Run `@vh/app-readme` to regenerate `README.md` locally               |
| `pnpm run test:static`      | ESLint + Prettier check across all workspaces (always runs in CI)    |
| `pnpm run test:types`       | TypeScript type check across all workspaces (always runs in CI)      |
| `pnpm run test:unit`        | Jest unit tests (`@vh/profile`, `@vh/quality-readme`)                |
| `pnpm run test:e2e`         | Playwright E2E suite (`@vh/quality-resume`)                          |
| `pnpm run build`            | Production build across all workspaces                               |
| `pnpm run test`             | Full pipeline: cleanup → build → static → types → unit → e2e         |
| `pnpm run test:doctor`      | Per-workspace validation gate used by the dependency bumper          |
| `pnpm run bumpDependencies` | Upgrade dependencies with NCU doctor mode (auto-rollback on failure) |

Script names are identical across all contexts (local, hooks, CI, NCU). See `docs/quality-gates.md`.

## CI/CD Pipeline

### Continuous Integration (Pull Requests targeting `main`)

Path filters detect which workspaces changed. Two jobs always run regardless:

- `test:static` — ESLint + Prettier across all workspaces
- `test:types` — TypeScript check across all workspaces

Conditional jobs (only when relevant paths changed):

| Job         | Triggers when                                                          |
| ----------- | ---------------------------------------------------------------------- |
| `test:unit` | `packages/profile` or `apps/readme` changed                            |
| `build`     | `apps/resume`, `packages/design-system`, or `packages/profile` changed |
| `test:e2e`  | After `build` completes (uses uploaded build artifact)                 |

See [`.github/workflows/ci.yml`](.github/workflows/ci.yml).

### Continuous Deployment (Push to `main`)

Every push to `main` is a release — CD tags automatically from root `package.json → version`. Three parallel post-build jobs handle tagging, GitHub Pages deploy, and README regeneration. See [`.github/workflows/cd.yml`](.github/workflows/cd.yml).

## Supply Chain Security

Configured in `pnpm-workspace.yaml` — read that file for current values, never rely on memory:

- **Minimum release age window** — blocks newly published packages, closing the most common malicious hijacking attack vector.
- **`allowBuilds`** — explicit whitelist of packages permitted to run postinstall scripts. Every entry documents the trust rationale as an inline comment. Packages not listed default to pnpm's global policy.
- **`catalog:`** — all dependency versions are centralized here. Individual `package.json` files reference `catalog:` instead of pinning versions directly.

To add a dependency: add it to `catalog:` in `pnpm-workspace.yaml`, then reference `catalog:` from
the workspace `package.json`. Never pin a version inside a workspace `package.json`.
